# v0.4 Admin Guide

## Scope and authorization

Admin pages are presentation only. Every Admin GET, PUT and DELETE route independently verifies a fresh Supabase claim and accepts only `app_metadata.role=admin`. `user_metadata`, email lists, query parameters and browser storage never grant Admin access. A changed app role takes effect only after a refreshed JWT or a new sign-in.

## Content workflow

Knowledge and cases use four states: `draft`, `reviewed`, `published` and `archived`.

- New records must start as `draft`.
- `draft → draft` allows normal draft edits; `draft → reviewed` starts review.
- `reviewed → reviewed` is intentionally allowed for necessary review corrections; it may return to `draft` or move to `published`.
- `published → archived` is the only published transition. Published payloads cannot be changed in place.
- `archived → draft` is the only archive transition. Edit only after the record is back in draft.
- Soft-deleted records are excluded from the normal Admin list and from public APIs. A second delete returns not found and cannot create another audit event.

The API applies these rules before a request is sent to the database. Migration `20260719_wisdom_os_admin_audit_hardening.sql` repeats them in database triggers, so direct Data API writes cannot bypass the workflow.

## Canonical system seed boundary

The migration has one narrow, separate maintenance path for `service_role` only. It can insert a missing canonical knowledge or case row only as active `published` content with both actor columns null. The browser, an authenticated user, `user_metadata`, and any supplied actor value cannot enter this path. Existing canonical system rows with byte-equivalent JSON are treated as no-ops; drift, Admin-managed ownership, non-published status, and soft deletion are rejected rather than overwritten. It cannot create drafts, modify content, revive deleted rows, or delete anything.

## Atomic audit trail

The same migration derives `create`, `update`, `status_transition` and `soft_delete` audit events in an `AFTER` trigger. A canonical system insert derives only `system_seed_create`, has a null actor, and uses fixed safe metadata. The content mutation and its audit insert share one database transaction: an audit failure rolls back the content mutation. The browser cannot provide the action or actor. Audit metadata is strictly limited to safe status/version markers and never stores payloads, credentials, request bodies, SQL or provider diagnostics.

`docs/v0.4-admin-audit-verification.sql` is read-only verification SQL. Its rollback notes are deliberately commented and require separate maintenance approval; they never delete content or audit history.

## Current validation boundary

The Admin workflow hardening is complete in local code and mock tests. The new migration has **not** been applied to Preview, and real Admin role, content mutation and audit-log browser acceptance have **not** been performed. Admin remains disabled by default; Production flags and credentials remain unset.
