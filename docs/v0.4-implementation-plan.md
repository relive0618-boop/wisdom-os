# Wisdom OS v0.4 實作計畫

## 現有資料與相容性

- 報告：`wisdom_reports_v1`，每筆為 `StoredReport`（`AnalyzeResponse` 加上 `createdAt`）。
- PDCA：`wisdom_pdca_cycle_<cycleId>`，每輪為獨立 `PdcaCycle`。
- v0.1 遷移：`sessionStorage.lastReport` 與舊 `wisdom_pdca_<title>` key 已保留安全遷移流程。
- 知識與案例：目前由 `knowledge.json`、`cases.json` 提供，API 與頁面可在沒有雲端服務時正常工作。

## 新增資料表

- `profiles`：使用者顯示名稱。
- `user_reports`、`user_pdca_cycles`：使用者私有報告與 PDCA payload、revision、soft delete 與同步 metadata。
- `knowledge_entries`、`case_entries`：版本化內容與發布狀態。
- `admin_audit_logs`：不含 payload 的管理操作紀錄。
- `rate_limit_buckets`：僅儲存 HMAC 雜湊識別碼的持久化限制桶。

## Auth 與 RLS

- Next.js `proxy.ts` 更新 Auth cookies；受保護的 Server Components 與 Route Handlers 使用 `getClaims()` 取得可信使用者與 app metadata。
- 使用者資料採 `auth.uid() = user_id` RLS；公開內容僅允許 `published` 且未 soft delete。
- 管理權限僅依 `app_metadata.role = admin`；不信任 `user_metadata`、request body 或 client role。
- service/secret client 僅供 rate limit、seed 與受保護維護工作，禁止一般 CRUD 與 client import。

## 同步策略

- 預設為 Local；登入並主動開啟同步後才為 Hybrid。
- Local repository 繼續使用現有 Zod 驗證與 localStorage；Cloud repository 只經過受保護 API。
- 每實體保存 revision、hash、最後同步時間與 sync state；雲端不可用時維持 Local。
- 比對 cloud revision、更新時間與 hash。雙方均變更時建立 conflict，不靜默覆寫。
- 衝突可保留本機、保留雲端或保留兩份；保留兩份會建立新 reportId/cycleId。

## 本機資料遷移

- 登入後顯示 migration wizard，先掃描並顯示數量。
- 使用者選擇才以每批最多 25 筆上傳；成功不刪除本機資料。
- 遷移進度與 pending 狀態保存於本機，允許中斷與續傳。

## Feature flags

- `NEXT_PUBLIC_WISDOM_CLOUD_SYNC_ENABLED=false`
- `NEXT_PUBLIC_WISDOM_ADMIN_ENABLED=false`
- `WISDOM_PERSISTENT_RATE_LIMIT_ENABLED=false`

Supabase URL 或 publishable key 缺失時不建立 client、不嘗試連線，UI 顯示雲端未啟用，v0.3 Local 模式維持不變。

## 測試策略

- Supabase client、claims、cloud API 與 repositories 採依賴注入與 fake client 測試，不使用真實 Key。
- 覆蓋 auth guards、RLS 設計、sync conflict、migration batching、cloud fallback、admin authorization、持久化 rate limit 與敏感資訊防護。
- Playwright 覆蓋 Local-only、未配置雲端、模擬登入、migration wizard、sync/conflict 與 admin mock flow。
