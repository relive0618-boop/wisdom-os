# v0.4 Preview 真实验收

本流程只针对 Vercel Preview；Production 的云端 credentials 与 feature flags 保持未设定。不要把任何 Secret、JWT、cookie 或连接资讯贴进 Git、PR、聊天、截图或 log。环境变量变更后必须重新部署 Preview。

## A. Preview 环境变量

仅在 Preview 设置以下名称：

```text
NEXT_PUBLIC_SUPABASE_URL
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY
SUPABASE_SECRET_KEY
NEXT_PUBLIC_WISDOM_CLOUD_SYNC_ENABLED
NEXT_PUBLIC_WISDOM_ADMIN_ENABLED
WISDOM_PERSISTENT_RATE_LIMIT_ENABLED
RATE_LIMIT_HASH_SECRET
```

URL 与 Publishable Key 可进入浏览器。`SUPABASE_SECRET_KEY` 与 `RATE_LIMIT_HASH_SECRET` 仅能存在服务器环境，绝不能使用 `NEXT_PUBLIC_` 前缀。Production 保持 false／unset。

## B. 公开 smoke test

```bash
pnpm verify:preview -- --base-url <PREVIEW_URL>
```

验收 health、知识库 56 条、案例 30 条、未登入 cloud API 拒绝、未登入 admin 拒绝，以及 response 不含 Secret。此工具只发送 GET，不建立帐号、不登入也不写入任何资料。

`--` 是 pnpm script 的参数分隔符；verifier 只会安全忽略最前方的单一分隔符，其他位置或重复的分隔符会被拒绝。若 Preview 受 Vercel Deployment Protection 保护，使用：

```bash
pnpm verify:preview -- --base-url <PREVIEW_URL> --vercel-protected
```

该模式使用当前登录 Vercel CLI 的官方 `vercel curl` 访问固定的只读 endpoint。它始终显式指定 Preview deployment，只发送 GET，不创建、保存、显示或手动传递任何 bypass 凭证。

## D. Cloud Sync 真实测试

Account A 在本机建立一份报告和至少一轮 PDCA，进入 `/sync` 后明确开始同步。确认云端写入成功而 localStorage 原资料仍在；重新整理与登出都不能删除本地资料。再以第二浏览器或无痕登入 Account A，pull 后确认完整资料。对同一资料制造冲突，并验证保留本地、保留云端、两者保留三种选择都不会自动删除版本。

- 在全新浏览器环境中，云端一份报告加一轮 PDCA 显示可安全下载两笔；无效资料为零。

## E. RLS 跨帐户测试

### 已完成的 Account A／B RLS 隔离验收

- Account A 保持一份云端报告与一轮 PDCA；Account B 的报告与 PDCA 清单均为零。
- Account B 无法列出或读取 Account A 的 report／PDCA；回应不含 payload、帐号资料、ID、凭证或服务端诊断资讯。
- 仅以 Account A 建立一份临时报告与一轮临时 PDCA；Account B 对实际更新端点与删除端点的请求均被拒绝。
- 临时资料由 Account A 清除后，Account A 回到一份报告与一轮 PDCA，Account B 仍为零。
- Account A 原有资料、Production、migration、RLS、Policies 与 Grants 均未修改。

验收记录不包含帐号、UUID、资料 ID、报告／PDCA 内容或任何凭证。

## F. Admin 权限测试

一般 Account A 不能进入 admin。不得信任 `user_metadata.role`；只能由 Supabase 安全管理界面设定 `app_metadata.role=admin`，并在重新登入或刷新 JWT 后生效。Admin 可管理 knowledge/cases，非 Admin 写入必须被拒绝；审计记录不得含 Secret、JWT、cookie 或完整 request body。

### 已执行的 Admin／Audit Preview 验收

`20260719_wisdom_os_admin_audit_hardening.sql` 已套用到 Preview，并完成只读验证：四个 trigger、两个固定 search path 的 `SECURITY DEFINER` function、function grants、既有 table grants、audit safe metadata 与既有 canonical content 都通过；migration 没有产生 audit event。canonical seed 的只读 preflight 已确认 knowledge 56／56、cases 30／30 为 canonical／稳定 JSON 语义等价，远端写入为 0。

独立真人验收以既有 Account B 完成。普通角色对 knowledge、cases 和 Audit Admin API 均为 403；仅临时新增 `app_metadata.role=admin` 并重新登入取得新 JWT 后，Admin 列表为 knowledge 56、cases 30、Audit 0。验收确认 draft-only create、draft／reviewed 编辑、合法状态转换、direct published create 拒绝、published／archived 原地修改拒绝、单次软删除与第二次删除 404。两笔临时内容最后保持软删除，公开内容维持 knowledge 56、cases 30。Audit 由 0 增为 10：create 2、update 2、status_transition 4、soft_delete 2；unsafe metadata 为 0，失败 mutation 均为零写入。角色移除并重新登入后，三条 Admin API 再次全部为 403。成功 mutation／Audit coupling 为 PASS；拒绝 mutation 零写入为 PASS；破坏性 audit-failure injection 未在真人环境执行，继续由自动化覆盖。

## G. Persistent Rate Limit

启用 Preview 的 persistent rate limit 后，对 `/api/analyze` 进行受控、低频的连续测试。超额应回传 429 与 `RATE_LIMITED`；数据库只保存 HMAC hash，不能保存原始 IP。RPC 故障时必须安全降级为记忆体限流，不能使 API 无限制，也不要进行高频压力测试。
